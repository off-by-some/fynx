    Stores - Reactive State Management¶

    Stores are FynX's answer to the fundamental question: "How do I organize my application's state?" They provide a structured, class-based approach to grouping related reactive values, computed properties, and methods into cohesive, reusable units. Think of stores as self-contained modules of state—each one representing a distinct domain in your application.

    Understanding Stores¶

    Before diving into code, let's understand what stores actually do and why they matter.

    The Problem Stores Solve¶

    As applications grow, state management becomes challenging. You might start with a few scattered observables, but soon you have dozens spread across your codebase. Questions arise: Which observables relate to each other? Where should state-modifying logic live? How do you test different parts of your state in isolation?

    Stores solve these problems by providing organizational structure. They group related state together, define clear boundaries between different domains, and encapsulate both data and the operations that modify it.

    What Makes a Store¶

    A Store is simply a class that inherits from Store and uses FynX's reactive primitives:

    • Observable attributes - The mutable state your application tracks
    • Computed properties - Derived values that automatically update when dependencies change
    • Reactive methods - Functions that run automatically in response to state changes
    • Regular methods - Actions that modify state or perform operations

    The Store base class provides additional functionality like subscription management, state snapshots, and reactive context handling, but at its core, a store is just a convenient way to organize your reactive code.

    Your First Store¶

    Let's build a store step by step, starting simple and progressively adding complexity.

    Step 1: Basic State¶

    The simplest store holds observable values:

    from fynx import Store, observable

class CounterStore(Store):
    count = observable(0)
    step = observable(1)

    This creates a store with two reactive attributes. The observable() function creates a descriptor that makes the attribute reactive—when you assign to CounterStore.count, all dependencies automatically update.

    # Reading values
current = CounterStore.count.value  # Access the current value

# Setting values
CounterStore.count = 5  # Assignment triggers reactivity
CounterStore.step = 2

    Notice the asymmetry: reading uses .value, but writing uses direct assignment. This is intentional—assignment syntax is more natural for updates, while .value makes it clear you're accessing a reactive observable's current state.

    Step 2: Adding Computed Properties¶

    Now let's add derived state using the >> operator, which creates computed observables:

    class CounterStore(Store):
    count = observable(0)
    step = observable(1)

    # Computed property - automatically updates when count changes
    doubled = count >> (lambda c: c * 2)
    is_positive = count >> (lambda c: c > 0)

    The >> operator takes an observable on the left and a transformation function on the right, creating a new computed observable. When count changes, doubled and is_positive automatically recalculate—but only when accessed. This lazy evaluation means computed properties have zero overhead until you actually need them.

    CounterStore.count = 5
print(CounterStore.doubled)      # 10 (computes now)
print(CounterStore.is_positive)  # True

CounterStore.count = -3
print(CounterStore.doubled)      # -6
print(CounterStore.is_positive)  # False

    Step 3: Methods for State Modification¶

    Stores become more useful when they encapsulate the logic for modifying their own state:

    class CounterStore(Store):
    count = observable(0)
    step = observable(1)

    doubled = count >> (lambda c: c * 2)
    is_positive = count >> (lambda c: c > 0)

    @classmethod
    def increment(cls):
        cls.count = cls.count.value + cls.step.value

    @classmethod
    def decrement(cls):
        cls.count = cls.count.value - cls.step.value

    @classmethod
    def reset(cls):
        cls.count = 0

    Now all counter logic lives in one place. Users of your store don't manipulate state directly—they call methods that express intent:

    CounterStore.increment()  # count becomes 1
CounterStore.increment()  # count becomes 2
CounterStore.step = 5
CounterStore.increment()  # count becomes 7
CounterStore.reset()      # count becomes 0

    This encapsulation makes your code more maintainable and testable. The store owns its state and defines how that state can change.

    Working with Multiple Observables¶

    Real applications need to coordinate multiple pieces of state. FynX provides the | operator to merge observables and the >> operator to transform the merged result.

    Combining Observables with |¶

    The | operator merges multiple observables into a tuple:

    class UserStore(Store):
    first_name = observable("Alice")
    last_name = observable("Smith")
    age = observable(30)

    # Merge first_name and last_name, then transform
    full_name = (first_name | last_name) >> (
        lambda first, last: f"{first} {last}"
    )

    # Merge all three for a complete description
    description = (first_name | last_name | age) >> (
        lambda first, last, years: f"{first} {last}, {years} years old"
    )

    When you merge observables with |, the transformation function receives one argument per observable, in order:

    UserStore.first_name = "Bob"
UserStore.last_name = "Johnson"
UserStore.age = 25

print(UserStore.full_name)     # "Bob Johnson"
print(UserStore.description)   # "Bob Johnson, 25 years old"

    Any change to a merged observable triggers recomputation of the entire result. This makes the | operator perfect for derived values that depend on multiple independent pieces of state.

    Practical Example: Shopping Cart¶

    Let's build something more realistic—a shopping cart with tax calculation:

    class ShoppingCartStore(Store):
    items = observable([])
    tax_rate = observable(0.08)  # 8% tax

    # Count total items
    item_count = items >> (
        lambda items_list: sum(item["quantity"] for item in items_list)
    )

    # Calculate subtotal
    subtotal = items >> (
        lambda items_list: sum(
            item["price"] * item["quantity"] 
            for item in items_list
        )
    )

    # Calculate tax amount
    tax_amount = (subtotal | tax_rate) >> (
        lambda sub, rate: sub * rate
    )

    # Calculate final total
    total = (subtotal | tax_amount) >> (
        lambda sub, tax: sub + tax
    )

    @classmethod
    def add_item(cls, name, price, quantity=1):
        """Add an item to the cart."""
        # Find if item already exists
        existing = next(
            (item for item in cls.items.value if item["name"] == name),
            None
        )

        if existing:
            # Update quantity of existing item
            cls.items = [
                {**item, "quantity": item["quantity"] + quantity}
                if item["name"] == name else item
                for item in cls.items.value
            ]
        else:
            # Add new item
            new_item = {"name": name, "price": price, "quantity": quantity}
            cls.items = cls.items.value + [new_item]

    @classmethod
    def remove_item(cls, name):
        """Remove an item from the cart."""
        cls.items = [
            item for item in cls.items.value 
            if item["name"] != name
        ]

    @classmethod
    def clear(cls):
        """Empty the cart."""
        cls.items = []

    Notice how computed properties chain naturally. total depends on subtotal and tax_amount, and tax_amount depends on subtotal and tax_rate. FynX tracks all these dependencies automatically:

    ShoppingCartStore.add_item("Widget", 10.99, 2)
ShoppingCartStore.add_item("Gadget", 15.50, 1)

print(f"Items: {ShoppingCartStore.item_count}")          # 3
print(f"Subtotal: ${ShoppingCartStore.subtotal:.2f}")    # $37.48
print(f"Tax: ${ShoppingCartStore.tax_amount:.2f}")       # $3.00
print(f"Total: ${ShoppingCartStore.total:.2f}")          # $40.48

# Change tax rate - total automatically updates
ShoppingCartStore.tax_rate = 0.10
print(f"New total: ${ShoppingCartStore.total:.2f}")      # $41.23

    Reactive Methods¶

    Sometimes you need side effects to happen automatically when state changes. The @reactive decorator makes methods execute whenever their dependencies update.

    Basic Reactive Methods¶

    from fynx import reactive

class NotificationStore(Store):
    unread_count = observable(0)
    user_name = observable("Guest")

    @staticmethod
    @reactive(unread_count, user_name)
    def update_page_title(count, name):
        """Update browser title when unread count or name changes."""
        if count > 0:
            title = f"({count}) Messages - {name}"
        else:
            title = f"Messages - {name}"
        # In a real app, this would update document.title
        print(f"Title: {title}")

    The @reactive decorator takes observables as arguments and calls the decorated function whenever any of them change. The function receives the current values as arguments:

    NotificationStore.user_name = "Alice"
# Prints: Title: Messages - Alice

NotificationStore.unread_count = 3
# Prints: Title: (3) Messages - Alice

NotificationStore.unread_count = 0
# Prints: Title: Messages - Alice

    Reactive methods are perfect for logging, analytics, UI updates, or any side effect that should happen in response to state changes.

    When to Use Reactive Methods vs Computed Properties¶

    This is a common point of confusion, so let's clarify:

    Use computed properties when you need a derived value that other code will read. Computed properties are lazy—they only calculate when accessed.

    Use reactive methods when you need to perform a side effect. Reactive methods are eager—they run immediately when dependencies change, regardless of whether anything reads their result.

    class ExampleStore(Store):
    count = observable(0)

    # Computed: lazy, returns a value, no side effects
    doubled = count >> (lambda c: c * 2)

    # Reactive: eager, performs side effects, return value ignored
    @staticmethod
    @reactive(count)
    def log_changes(new_count):
        print(f"Count changed to {new_count}")

    Advanced Patterns¶

    Once you're comfortable with basics, these patterns will help you build more sophisticated applications.

    Conditional Computed Properties¶

    You can create computed properties that branch based on state:

    class TodoStore(Store):
    todos = observable([])
    filter_mode = observable("all")  # "all", "active", "completed"

    # Always compute these
    active_todos = todos >> (
        lambda todos: [t for t in todos if not t["completed"]]
    )
    completed_todos = todos >> (
        lambda todos: [t for t in todos if t["completed"]]
    )

    # Dynamically filter based on mode
    visible_todos = (todos | filter_mode) >> (
        lambda todos, mode: {
            "all": todos,
            "active": [t for t in todos if not t["completed"]],
            "completed": [t for t in todos if t["completed"]]
        }[mode]
    )

    This pattern lets you change how data is filtered without redefining computed properties:

    TodoStore.todos = [
    {"id": 1, "text": "Learn FynX", "completed": False},
    {"id": 2, "text": "Build app", "completed": True},
    {"id": 3, "text": "Write tests", "completed": False}
]

TodoStore.filter_mode = "active"
print(len(TodoStore.visible_todos))  # 2

TodoStore.filter_mode = "completed"
print(len(TodoStore.visible_todos))  # 1

    Chaining Computed Properties¶

    Computed properties can depend on other computed properties, creating transformation pipelines:

    class DataAnalysisStore(Store):
    raw_data = observable([10, 20, 30, 40, 50])

    # Level 1: Basic statistics
    count = raw_data >> len
    total = raw_data >> sum
    mean = (total | count) >> (
        lambda total, n: total / n if n > 0 else 0
    )

    # Level 2: Variance (depends on mean)
    variance = (raw_data | mean | count) >> (
        lambda data, avg, n: (
            sum((x - avg) ** 2 for x in data) / (n - 1)
            if n > 1 else 0
        )
    )

    # Level 3: Standard deviation (depends on variance)
    std_dev = variance >> (lambda var: var ** 0.5)

    # Level 4: Coefficient of variation (depends on std_dev and mean)
    cv = (std_dev | mean) >> (
        lambda std, avg: (std / avg * 100) if avg != 0 else 0
    )

    Each level builds on the previous one. When raw_data changes, FynX propagates the update through all dependent computed properties in the correct order:

    print(f"Mean: {DataAnalysisStore.mean:.2f}")
print(f"Std Dev: {DataAnalysisStore.std_dev:.2f}")
print(f"CV: {DataAnalysisStore.cv:.2f}%")

DataAnalysisStore.raw_data = [5, 10, 15, 20, 25]
# All statistics automatically recalculate

    Cross-Store Dependencies¶

    Stores can depend on other stores, enabling modular state management:

    class UserPreferencesStore(Store):
    theme = observable("light")
    language = observable("en")
    font_size = observable(14)

class UIStore(Store):
    current_page = observable("home")

    # Computed properties that depend on another store
    page_title = (current_page | UserPreferencesStore.language) >> (
        lambda page, lang: {
            "en": {"home": "Home", "settings": "Settings"},
            "es": {"home": "Inicio", "settings": "Configuración"},
            "fr": {"home": "Accueil", "settings": "Paramètres"}
        }.get(lang, {"home": "Home", "settings": "Settings"})[page]
    )

    css_vars = (UserPreferencesStore.theme | UserPreferencesStore.font_size) >> (
        lambda theme, font: {
            "--bg-color": {"light": "#ffffff", "dark": "#1a1a1a"}.get(theme, "#ffffff"),
            "--text-color": {"light": "#000000", "dark": "#ffffff"}.get(theme, "#000000"),
            "--font-size": f"{font}px"
        }
    )

    This pattern keeps stores focused on their own domains while still allowing them to coordinate:

    UIStore.current_page = "settings"
UserPreferencesStore.language = "es"
print(UIStore.page_title)  # "Configuración"

UserPreferencesStore.theme = "dark"
print(UIStore.css_vars)
# {"--bg-color": "#1a1a1a", "--text-color": "#ffffff", "--font-size": "14px"}

    Best Practices¶

    Organize by Domain¶

    Create separate stores for distinct parts of your application:

    # Good: Clear separation of concerns
class AuthStore(Store):
    user = observable(None)
    is_authenticated = observable(False)
    token = observable(None)

class CartStore(Store):
    items = observable([])
    checkout_status = observable("idle")

class UIStore(Store):
    sidebar_open = observable(False)
    modal_content = observable(None)

    Avoid creating one giant store that holds everything—it defeats the purpose of modular organization.

    Keep Methods Pure When Possible¶

    Methods that modify state should be straightforward and predictable:

    class TodoStore(Store):
    todos = observable([])

    @classmethod
    def add_todo(cls, text):
        """Good: Simple, predictable, no hidden side effects."""
        new_todo = {
            "id": str(hash(text + str(len(cls.todos.value)))),
            "text": text,
            "completed": False
        }
        cls.todos = cls.todos.value + [new_todo]

    @classmethod
    def toggle_todo(cls, todo_id):
        """Good: Single responsibility, clear intent."""
        cls.todos = [
            {**t, "completed": not t["completed"]}
            if t["id"] == todo_id else t
            for t in cls.todos.value
        ]

    Treat Observables as Immutable¶

    Always create new values rather than mutating existing ones:

    # Good: Create a new list
TodoStore.todos = TodoStore.todos.value + [new_todo]

# Bad: Mutate the existing list (won't trigger reactivity!)
TodoStore.todos.value.append(new_todo)

    FynX detects changes through assignment. If you mutate an observable's value in place, dependencies won't update because from FynX's perspective, nothing changed.

    Handle Edge Cases in Computed Properties¶

    Computed properties should be defensive about their inputs:

    class StatsStore(Store):
    values = observable([])

    # Good: Handles empty list gracefully
    average = values >> (
        lambda vals: sum(vals) / len(vals) if len(vals) > 0 else 0
    )

    # Good: Handles division by zero
    ratio = (numerator | denominator) >> (
        lambda num, denom: num / denom if denom != 0 else 0
    )

    Computed properties can't raise exceptions—if they do, they break the reactive chain. Always handle edge cases explicitly.

    Test Stores in Isolation¶

    Stores are easy to test because they encapsulate state:

    def test_counter_store():
    # Reset to known state
    CounterStore.count = 0
    CounterStore.step = 1

    # Test operations
    CounterStore.increment()
    assert CounterStore.count.value == 1
    assert CounterStore.doubled == 2

    CounterStore.step = 5
    CounterStore.increment()
    assert CounterStore.count.value == 6

    CounterStore.reset()
    assert CounterStore.count.value == 0

    Complete Example: Todo Application¶

    Here's a comprehensive example that demonstrates all the concepts we've covered:

    from fynx import Store, observable, computed, reactive

class TodoApplicationStore(Store):
    """Complete todo application with filtering, statistics, and status updates."""

    # Core state
    todos = observable([])
    filter_mode = observable("all")  # "all", "active", "completed"

    # Filtered views
    active_todos = todos >> (
        lambda todos: [t for t in todos if not t["completed"]]
    )
    completed_todos = todos >> (
        lambda todos: [t for t in todos if t["completed"]]
    )

    # Statistics
    total_count = todos >> len
    active_count = active_todos >> len
    completed_count = completed_todos >> len

    # Status checks
    is_empty = total_count >> (lambda c: c == 0)
    all_completed = (active_count | total_count) >> (
        lambda active, total: total > 0 and active == 0
    )
    has_completed = completed_count >> (lambda c: c > 0)

    # Dynamic filtering based on current mode
    visible_todos = (todos | filter_mode) >> (
        lambda todos, mode: {
            "all": todos,
            "active": [t for t in todos if not t["completed"]],
            "completed": [t for t in todos if t["completed"]]
        }.get(mode, todos)
    )

    # Human-readable status message
    status_message = observable("")

    @staticmethod
    @reactive(is_empty, all_completed, active_count, completed_count)
    def _update_status_message(empty, all_done, active, completed):
        """Automatically update status message when counts change."""
        if empty:
            msg = "No todos yet. Add one to get started!"
        elif all_done:
            msg = f"🎉 All {completed} todos completed!"
        elif completed == 0:
            plural = "todo" if active == 1 else "todos"
            msg = f"{active} active {plural}"
        else:
            msg = f"{active} active, {completed} completed"

        TodoApplicationStore.status_message.set(msg)

    # Public API
    @classmethod
    def add_todo(cls, text):
        """Add a new todo with the given text."""
        if not text.strip():
            return

        new_todo = {
            "id": str(hash(text + str(len(cls.todos.value)))),
            "text": text.strip(),
            "completed": False
        }
        cls.todos = cls.todos.value + [new_todo]

    @classmethod
    def toggle_todo(cls, todo_id):
        """Toggle the completed status of a todo."""
        cls.todos = [
            {**t, "completed": not t["completed"]}
            if t["id"] == todo_id else t
            for t in cls.todos.value
        ]

    @classmethod
    def update_todo_text(cls, todo_id, new_text):
        """Update the text of a todo."""
        if not new_text.strip():
            return

        cls.todos = [
            {**t, "text": new_text.strip()}
            if t["id"] == todo_id else t
            for t in cls.todos.value
        ]

    @classmethod
    def delete_todo(cls, todo_id):
        """Remove a todo from the list."""
        cls.todos = [
            t for t in cls.todos.value 
            if t["id"] != todo_id
        ]

    @classmethod
    def clear_completed(cls):
        """Remove all completed todos."""
        cls.todos = [
            t for t in cls.todos.value 
            if not t["completed"]
        ]

    @classmethod
    def toggle_all(cls):
        """Toggle all todos to the opposite of the current majority state."""
        if cls.all_completed:
            # If all completed, mark all as active
            cls.todos = [
                {**t, "completed": False} 
                for t in cls.todos.value
            ]
        else:
            # Otherwise, mark all as completed
            cls.todos = [
                {**t, "completed": True} 
                for t in cls.todos.value
            ]

# Usage demonstration
TodoApplicationStore.add_todo("Learn FynX stores")
TodoApplicationStore.add_todo("Build a reactive app")
TodoApplicationStore.add_todo("Write comprehensive tests")

print(TodoApplicationStore.status_message.value)
# Output: 3 active todos

TodoApplicationStore.toggle_todo(TodoApplicationStore.todos.value[0]["id"])
print(TodoApplicationStore.status_message.value)
# Output: 2 active, 1 completed

TodoApplicationStore.filter_mode = "active"
print(f"Visible: {len(TodoApplicationStore.visible_todos)} todos")
# Output: Visible: 2 todos

TodoApplicationStore.toggle_all()
print(TodoApplicationStore.status_message.value)
# Output: 🎉 All 3 todos completed!

    Summary¶

    Stores provide structure for organizing reactive state in FynX applications. They combine observables, computed properties, and reactive methods into cohesive units that represent distinct domains of your application.

    Key takeaways:

    • Stores group related state - Keep observables that belong together in the same store
    • Observable attributes hold mutable state - Use observable() to make class attributes reactive
    • Computed properties derive new values - Use >> to create automatically updating derived state
    • The | operator merges observables - Combine multiple observables when computed values need multiple inputs
    • Reactive methods perform side effects - Use @reactive for automatic side effects when state changes
    • Methods encapsulate state changes - Define clear APIs for modifying your store's state
    • Stores can depend on other stores - Build modular applications with cross-store dependencies

    With stores, you can build maintainable, testable reactive applications that scale from simple counters to complex, multi-domain state management systems.
