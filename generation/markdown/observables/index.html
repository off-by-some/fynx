    Observables - The Heart of Reactive Programming¶

    Observables are the fundamental building blocks of reactive programming in FynX. They represent values that can change over time and automatically notify anything that depends on them. Think of observables as "reactive variables" that trigger updates throughout your application whenever their values change.

    What are Observables?¶

    At their core, observables are containers for values that:

    • Hold data that can change over time
    • Notify subscribers when their values change
    • Enable reactive relationships between different parts of your application
    • Support composition through operators like |, >>, &, and ~

    Unlike regular variables, observables create a reactive graph where changes propagate automatically through your entire application.

    Creating Observables¶

    Basic Observable Creation¶

    from fynx import observable

# Create observables with initial values
user_name = observable("Alice")
age = observable(30)
is_online = observable(False)

    Reading and Writing Observable Values¶

    # Reading values - use .value property
current_name = user_name.value
current_age = age.value

# Writing values - direct assignment
user_name = "Bob"  # Triggers reactivity
age = 31           # Triggers reactivity
is_online = True   # Triggers reactivity

    Reactivity Fundamentals¶

    Subscribing to Changes¶

    Observables notify subscribers whenever their values change:

    def on_name_change(new_name):
    print(f"Name changed to: {new_name}")

# Subscribe to changes
user_name.subscribe(on_name_change)

user_name = "Charlie"  # Prints: "Name changed to: Charlie"
user_name = "Diana"    # Prints: "Name changed to: Diana"

# Unsubscribe when no longer needed
user_name.unsubscribe(on_name_change)
user_name = "Eve"  # No output - unsubscribed

    Reactive Context Management¶

    Observables track which code depends on them during execution:

    # This function creates a dependency on user_name
def greet_user():
    name = user_name.value  # Creates dependency
    return f"Hello, {name}!"

# The function will automatically re-run when user_name changes
user_name = "Frank"
# greet_user() would now return "Hello, Frank!" if called again

    Combining Observables¶

    Merging with the | Operator¶

    Combine multiple observables into reactive tuples:

    # Merge two observables
name_and_age = user_name | age

def on_user_data_change(name, age):
    print(f"User: {name}, Age: {age}")

# Subscribe to the merged observable
name_and_age.subscribe(on_user_data_change)

user_name = "Grace"  # Prints: "User: Grace, Age: 31"
age = 32            # Prints: "User: Grace, Age: 32"

    Creating Computed Values with >>¶

    Transform observable values reactively:

    # Create derived values
full_name = (user_name | age) >> (
    lambda name, age_val: f"{name} (age {age_val})"
)

is_adult = age >> (lambda a: a >= 18)
is_senior = age >> (lambda a: a >= 65)

# Subscribe to computed values
full_name.subscribe(lambda val: print(f"Full name: {val}"))
is_adult.subscribe(lambda val: print(f"Is adult: {val}"))

user_name = "Henry"
age = 25
# Output:
# Full name: Henry (age 25)
# Is adult: True

    Conditional Filtering with &¶

    Only emit values when conditions are met:

    messages = observable(0)
user_online = observable(False)

# Only notify when user is online AND has messages
should_notify = messages & user_online

def send_notification(count):
    print(f"📬 Sending {count} messages!")

should_notify.subscribe(send_notification)

messages = 3        # No notification - user offline
user_online = True  # Now notifies: "📬 Sending 3 messages!"
messages = 5        # Notifies again: "📬 Sending 5 messages!"
user_online = False # No more notifications
messages = 10       # Still no notification - user offline

    Negation with ~¶

    Invert boolean observables:

    is_loading = observable(True)
is_idle = ~is_loading  # True when not loading

is_idle.subscribe(lambda idle: print(f"Idle: {idle}"))

is_loading = True   # Prints: "Idle: False"
is_loading = False  # Prints: "Idle: True"

    Advanced Observable Patterns¶

    Chained Transformations¶

    Build complex data pipelines:

    raw_scores = observable([65, 72, 88, 91, 77])

# Chain multiple transformations
average_score = raw_scores >> (
    lambda scores: sum(scores) / len(scores) if scores else 0
)

grade_letter = average_score >> (
    lambda avg: "A" if avg >= 90 else "B" if avg >= 80 else "C" if avg >= 70 else "F"
)

letter_count = grade_letter >> (
    lambda grade: len(grade)  # Always 1 for our grades
)

# Subscribe to see the full pipeline in action
average_score.subscribe(lambda val: print(f"Average: {val:.1f}"))
grade_letter.subscribe(lambda val: print(f"Grade: {val}"))

raw_scores = [85, 92, 78, 96, 88]
# Output:
# Average: 87.8
# Grade: B

    Multiple Subscriptions¶

    One observable can have many subscribers:

    counter = observable(0)

# Multiple subscribers to the same observable
counter.subscribe(lambda val: print(f"Counter: {val}"))
counter.subscribe(lambda val: print(f"📊 Analytics: {val}"))
counter.subscribe(lambda val: print(f"🔥 Hot value: {val}"))

counter = 1
# Output:
# Counter: 1
# 📊 Analytics: 1
# 🔥 Hot value: 1

    Observable Cleanup¶

    Properly manage subscriptions to prevent memory leaks:

    def setup_reactions():
    def log_changes(val):
        print(f"Value changed: {val}")

    # Always store subscription references for cleanup
    subscription = counter.subscribe(log_changes)

    # Later, when you no longer need the subscription
    counter.unsubscribe(log_changes)
    # or
    counter.unsubscribe(subscription)

    Observable Types in FynX¶

    FynX provides several specialized observable types:

    Regular Observables (Observable)¶

    The basic type for reactive values:

    from fynx import Observable

# Direct instantiation (less common)
name_obs = Observable("name", "Alice")
name_obs.subscribe(lambda val: print(f"Name: {val}"))

name_obs.set("Bob")  # Prints: "Name: Bob"

    Computed Observables (ComputedObservable)¶

    Values derived from other observables:

    from fynx import computed

# Using the computed function
total = computed(lambda a, b: a + b, x | y)

# Or using the >> operator (recommended)
total = (x | y) >> (lambda a, b: a + b)

    Merged Observables (MergedObservable)¶

    Combinations of multiple observables:

    # Created with the | operator
coordinates = x | y | z

def on_position_change(x_val, y_val, z_val):
    print(f"Position: ({x_val}, {y_val}, {z_val})")

coordinates.subscribe(on_position_change)

    Conditional Observables (ConditionalObservable)¶

    Filtered reactive streams:

    # Created with the & operator
valid_input = user_input & is_valid_length

valid_input.subscribe(lambda val: print(f"Valid: {val}"))

    Common Observable Patterns¶

    State Management¶

    class AppState:
    current_user = observable(None)
    is_authenticated = observable(False)
    theme = observable("light")

# Reactive UI components would subscribe to these

    Form Validation¶

    email = observable("")
password = observable("")

# Computed validation states
email_valid = email >> (
    lambda e: "@" in e and "." in e.split("@")[1]
)
password_valid = password >> (lambda p: len(p) >= 8)

# Combined form validity
form_valid = email_valid & password_valid

form_valid.subscribe(lambda valid: print(f"Form valid: {valid}"))

    API State¶

    # Loading states
is_loading = observable(False)
error_message = observable(None)
data = observable(None)

# Computed UI states
show_loading = is_loading
show_error = error_message >> (lambda msg: msg is not None)
show_data = (data >> (lambda d: d is not None)) & ~is_loading

    Event Handling¶

    click_count = observable(0)
last_click_time = observable(None)

# Rate limiting
recent_clicks = (click_count | last_click_time) >> (
    lambda count, time: count if time and (datetime.now() - time).seconds < 60 else 0
)

    Best Practices¶

    1. Use Meaningful Names¶

    # Good
user_authenticated = observable(False)
selected_items = observable([])

# Avoid
obs1 = observable(False)
data = observable([])

    2. Handle Edge Cases in Transformations¶

    # Good: Handles empty arrays
average = numbers >> (
    lambda nums: sum(nums) / len(nums) if nums else 0
)

# Good: Handles None values
display_name = user >> (
    lambda u: u["name"] if u and "name" in u else "Anonymous"
)

    3. Clean Up Subscriptions¶

    class Component:
    def __init__(self):
        self._subscriptions = []

        # Store subscription references
        sub = some_observable.subscribe(self._on_change)
        self._subscriptions.append(sub)

    def destroy(self):
        # Clean up all subscriptions
        for sub in self._subscriptions:
            some_observable.unsubscribe(sub)
        self._subscriptions.clear()

    4. Prefer >> Over Manual computed()¶

    # Preferred
result = (a | b) >> (lambda x, y: x + y)

# Less readable
result = computed(lambda x, y: x + y, a | b)

    5. Chain Computations Thoughtfully¶

    # Good: Clear step-by-step transformations
processed_data = raw_data >> validate >> transform >> format_output

# Avoid: Overly complex single transformations
processed_data = raw_data >> (
    lambda data: format_output(transform(validate(data)))
)

    Performance Considerations¶

    Lazy Evaluation¶

    Observables only compute values when accessed:

    expensive_calc = data >> (lambda d: slow_computation(d))

# slow_computation() only runs here, when accessed
result = expensive_calc.value

    Dependency Tracking¶

    FynX automatically tracks which observables are used:

    # Only tracks observables actually accessed during computation
result = (a | b | c) >> (
    lambda x, y, z: x + y  # Only x and y used, z ignored
)
# FynX knows result doesn't depend on c

    Memory Management¶

    Unused reactive contexts are automatically cleaned up, but explicit subscription management helps:

    # Always unsubscribe when components are destroyed
# Always use context managers for temporary subscriptions

    Observables are the foundation of reactive programming in FynX. They enable you to build applications where state changes automatically propagate through your entire system, creating responsive and maintainable user interfaces and business logic. Once you understand observables, the rest of FynX's features build naturally on top of this foundation.
